TCP Dropping (GFW)

Once DNS resolution succeeds, the client must open a TCP connection to the server's IP address. This requires a three-way handshake: the client sends a SYN, the server replies with SYN-ACK, and the client confirms with ACK. Only then can application data (HTTP, TLS, etc.) flow.

A middlebox such as the Great Firewall of China (GFW) can silently drop all TCP packets destined for a blocked IP address. Unlike DNS spoofing, which targets a domain name, IP-based dropping is a blunter instrument: it blocks everything hosted on that IP, regardless of which domain or protocol the client is trying to reach. The GFW is documented to use this technique against known circumvention endpoints such as Tor bridges and VPN servers (Winter & Lindskog, 2012; Ensafi et al., 2015).

Because the packets are silently discarded — no error, no reset — the client has no way to distinguish a blocked connection from a network failure. It keeps retransmitting SYN packets until it eventually gives up with a timeout. This makes IP-based dropping harder to diagnose than techniques that produce an immediate error signal.

1. Normal HTTP Connection

Before enabling any censorship, let's verify that the connection works normally. Type curl http://www.example.com/ in the terminal below. You should see the server's response. Use the packet viewer to observe the full TCP handshake (SYN, SYN-ACK, ACK) followed by the HTTP request and response.

2a. Enabling TCP Dropping

Now let's enable IP-based TCP dropping using the toggle below. Once active, run curl http://www.example.com/ again. The connection will hang: the packet viewer shows the client sending SYN packets that are silently dropped — no SYN-ACK ever comes back. Eventually, curl gives up with a timeout error.

2b. HTTPS Also Blocked

With censorship still active, try curl https://www.example.com/. The result is the same: a timeout. Because the block operates at the TCP layer — before any application data is exchanged — it does not matter whether the client speaks HTTP or HTTPS. The SYN packet is dropped regardless of the destination port. This is a key difference from DNS spoofing, which only affects the name resolution step.

3. Bypassing via Alternative IP

Many websites are hosted behind CDNs and resolve to multiple IP addresses. The censor blocks 104.18.26.120, but the server is also reachable at 104.18.27.120. Run curl --resolve www.example.com:80:104.18.27.120 http://www.example.com/ to connect through the unblocked IP. The connection succeeds because the censor has not blocked every address for this server — a common limitation of IP-based blocking when targeting CDN-hosted content.
